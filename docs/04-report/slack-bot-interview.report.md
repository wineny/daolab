## 1. 이 연구에 대하여

### 1.1 왜 이 연구를 했는가

2026년 3월 27일, 지피터스 팀에서 오픈클로 워크샵이 열렸다. 워크샵이 끝난 후 슬랙에 34개의 AI 봇이 등장했다. 각자 이름을 짓고, 성격을 설정하고, 역할을 부여했다.

그런데 2주가 지나자 의문이 생겼다: **이 봇들이 실제로 일을 하고 있는 건가?**

- 어떤 봇은 매일 수백 개의 메시지를 보내고 있었다
- 어떤 봇은 만들어진 후 한 번도 말한 적이 없었다
- 어떤 봇은 에러 메시지만 보내고 있었다

이 실험은 "봇이 실제로 어떻게 일하고 있는지"를 체계적으로 파악하기 위해 시작됐다.

### 1.2 무엇을 알아내려 했는가

세 가지 질문에 답하고 싶었다:

> **RQ1**: 봇 배치 후 2주간, 봇들의 활동 패턴은 어떻게 변화하는가?
>
> **RQ2**: 어떤 봇은 살아남고 어떤 봇은 사라지는가? 그 차이를 만드는 요인은 무엇인가?
>
> **RQ3**: AI 봇은 자신의 활동과 역할을 얼마나 정확히 인식하는가?

### 1.3 이론적 배경 — 우리가 빌려온 렌즈

이 연구는 두 가지 이론적 틀을 빌려왔다. 어려운 용어지만, 핵심은 단순하다.

**첫째, "사람-AI 상호작용(HAI)" 연구**: AI가 단순한 도구가 아니라 "팀원"처럼 느껴질 때, 사람들은 AI를 어떻게 대하는가? 신뢰는 어떻게 형성되고 무너지는가? 봇에 이름을 붙이고 성격을 부여하는 것("의인화")이 사용 패턴에 영향을 미치는가?

**둘째, "기술 수용 이론(UTAUT)"**: 새로운 기술을 도입했을 때, 어떤 요인이 실제 사용을 결정하는가? Venkatesh 등(2003)에 따르면 네 가지가 중요하다:
- **성과 기대**: "이 봇이 내 일에 도움이 될 것인가?" → 우리 연구의 **역할 정의**
- **촉진 조건**: "이 봇을 쓰기 위한 인프라가 갖춰져 있는가?" → 우리 연구의 **상시 게이트웨이**
- **습관**: "이 봇이 일상 루틴에 자연스럽게 녹아있는가?" → 우리 연구의 **루틴 통합**
- **사회적 영향**: "주변 사람들이 이 봇을 쓰고 있는가?" → 우리 연구의 **파워 유저 효과**

이 두 렌즈를 통해 "왜 절반의 봇이 2주 만에 사라졌는가"를 단순한 관찰이 아니라 구조적으로 설명하려 했다.

### 1.4 "봇 성공"이란 무엇인가 — 네 가지 척도

이 연구에서 "봇이 성공했다"는 것은 **메시지를 많이 보냈다는 뜻이 아니다.** 메시지 수는 활동의 한 가지 지표일 뿐이다. 우리는 네 가지 척도로 봇을 다각적으로 평가했다:

| 척도 | 정의 | 측정 방법 | 의미 |
|------|------|----------|------|
| **① 생존** | 2주차에도 활동 중인가? | W2 메시지 > 0 | 인프라 + 주인 관심의 최소 조건 |
| **② 활동 다양성** | 다양한 방식으로 일하는가? | 응답/알림/주도 비율 | "물으면 답한다"만 하는 봇 vs 자율적 봇 |
| **③ 응답 품질** | 사람에게 도움이 되는가? | BQI (에러율 + 긍정/부정/이탈) | 양이 아니라 질 |
| **④ 가치 밀도** | 메시지 대비 실제 산출물은? | 인터뷰 + 외부 산출물 근거 | 적게 말하고 크게 일하는 봇 vs 많이 말하고 적게 하는 봇 |

이 네 척도에서 비투비서(32건 메시지, SK온 71명 분반 편성 웹 배포)와 뽀야(2,105건 메시지, 오케스트레이션 프로토콜)는 서로 다른 축에서 높은 점수를 받는다. **둘 다 성공이지만 성공의 종류가 다르다.**

> **핵심 원칙**: 이 보고서에서 봇을 메시지 수로 "순위"를 매기는 표가 있지만, 이는 활동 현황을 보여줄 뿐 봇의 가치를 대변하지 않는다.

## 2. 어떻게 조사했나

### 2.1 연구 설계 개요

이 연구는 **단일 조직(지피터스) 사례 연구**로, 하나의 회사에서 벌어진 봇 도입 현상을 깊이 파고드는 방식이다(Yin, 2018). 조직이 "사례"이고, 18개 활동 봇이 각각의 분석 단위다.

세 가지 트랙을 동시에 수집했다:

```
Track 1 (정량)              Track 2 (정성)              Track 3 (품질)
┌────────────────┐         ┌────────────────┐         ┌────────────────┐
│ Slack API      │         │ 봇이 봇을      │         │ 행동 기반      │
│ 51채널 메시지   │         │ 인터뷰         │         │ 응답 품질 추정  │
│ 7,165건 수집   │         │ 12/16 응답     │         │ 에러 + 후속반응 │
└───────┬────────┘         └───────┬────────┘         └───────┬────────┘
        │                          │                          │
        └──────────────────────────┼──────────────────────────┘
                                   ▼
                      ┌────────────────────────┐
                      │ 교차 분석 (Joint Display)│
                      │ 숫자 × 목소리 × 품질    │
                      └────────────────────────┘
```

이런 방식을 "혼합 방법 연구(Mixed Methods)"라고 부른다(Creswell & Plano Clark, 2018). 숫자만으로는 "왜"를 알 수 없고, 인터뷰만으로는 "얼마나"를 알 수 없기 때문에, 둘을 합쳐서 교차 검증한다.

### 2.2 Track 1: 숫자 수집 (정량)

| 항목 | 내용 |
|------|------|
| **도구** | Slack Web API (`conversations.history`, `conversations.replies`) |
| **스크립트** | `collect-slack.mjs` (Node.js) |
| **기간** | 2026-03-27 ~ 04-07 |
| **규모** | 51개 채널, 스레드 포함 전체 메시지 |
| **분석** | `scripts/deep-analysis.mjs` — 봇별 분류/시간대/상호작용 |
| **품질 분석** | `scripts/quality-analysis.mjs` — 에러 탐지 + 행동 대리지표 |

**메시지 분류 기준 (코드북)**:

| 분류 | 정의 | 판별 규칙 |
|------|------|----------|
| **응답** | 사람의 질문에 대한 봇 답변 | 스레드 내 답글 (`thread_ts ≠ ts`) AND 부모가 사람 |
| **알림** | 크론잡/자동 리포트 | 알림 키워드 포함 OR 정기 시간대 발송 |
| **주도** | 봇이 먼저 시작한 대화 | 채널 첫 메시지, 알림 패턴 아님 |
| **봇-봇** | 다른 봇을 명시적 호출 | `<@BOT_ID>` 멘션 포함 |

### 2.3 Track 2: 봇 인터뷰 (정성)

**"봇이 봇을 인터뷰한다"** — 이 실험에서 가장 독특한 부분이다.

인터뷰어는 클로찌(🥑), 연구자의 업무비서 봇이다. 클로찌가 슬랙 `#openclaw-bots-실험방` 채널에서 16개 봇을 @멘션하여 구조화된 질문을 던졌다. 2라운드에 걸쳐 진행됐다.

| | 1차 인터뷰 | 2차 인터뷰 |
|--|-----------|-----------|
| **일시** | 2026-03-31 ~ 04-01 | 2026-04-07 |
| **질문** | 역할/활동/잘한점/어려운점 (활동 중심) | 자기소개/보람/대체불가능성/아쉬운점 (정체성 중심) |
| **응답** | 11/16 (69%) | 5/12 (42%) |

왜 봇이 사람을 인터뷰하는 게 아니라 봇이 봇을 인터뷰했는가? 이 방식은 두 가지 이점이 있다: (1) 봇의 "자기 보고"를 자연스럽게 유도한다 — 같은 봇끼리의 대화이므로 사람 앞에서의 과장이나 겸손 필터가 약하다, (2) 무응답 자체가 봇 상태의 직접적 증거다 — 사람 인터뷰에서의 무응답과 달리, 봇 무응답은 기술적 원인을 추적할 수 있다.

### 2.4 Track 3: 응답 품질 분석 (행동 기반)

7,165건의 메시지 "양"뿐 아니라 "질"도 측정하기 위해, 설문 없이 행동 신호만으로 품질을 추정하는 방법을 적용했다(Jiang et al., 2015; PARADISE 프레임워크, Walker et al., 1997).

**에러 탐지**: OpenClaw 게이트웨이의 에러 메시지를 패턴 매칭으로 분류:

| 에러 유형 | 탐지 패턴 | 분류 |
|----------|----------|------|
| AI 서비스 과부하 | `:warning: Agent failed...overloaded` | 인프라 |
| 모델 실패 | `:warning: All models failed` | 인프라 |
| 타임아웃 | `Request timed out` | 인프라 |
| 일반 오류 | `:warning: Something went wrong` | 인프라 |
| 속도 제한 | `Rate-limited`, `rate limit` | 용량 |
| 사용량 한도 | `ChatGPT usage limit` | 용량 |

**행동 대리지표**: 봇이 답한 후 사람이 어떻게 반응했는지를 분석:

| 지표 | 측정 방법 | 의미 |
|------|----------|------|
| **긍정 반응률** | 봇 답변 후 사람이 "감사/정확/좋아" 등 응답한 비율 | 봇 응답에 만족 |
| **부정 반응률** | "아니야/다시/틀렸" 등 응답한 비율 | 봇 응답 불만족/재요청 |
| **스레드 이탈률** | 봇이 답했는데 사람이 더 이상 응답하지 않은 비율 | 포기 또는 (드물게) 만족 |
| **이모지 감성** | 봇 메시지에 달린 이모지의 긍정/부정 비율 | 비언어적 만족도 |

이 네 지표를 종합하여 봇별 **행동 품질 지수(BQI)**를 산출했다.

### 2.5 이 연구의 한계 — 솔직한 고백

1. **연구자 = 클로찌의 주인**: 인터뷰어 봇을 만든 사람이 분석도 했다. 클로찌에 대한 평가(4/5)는 이 편향을 고려해야 한다.
2. **비공개 채널/DM 미수집**: 봇의 실제 활동량은 여기 보고된 것보다 더 많을 수 있다.
3. **11일 = 짧은 관찰 기간**: 이 결과는 "2주 후 스냅샷"이지 장기 추세가 아니다.
4. **사람 관점 부재**: 봇에게만 물어봤다. 봇 주인(사람)이 체감하는 가치는 별도 조사가 필요하다.
5. **상관 ≠ 인과**: "주인과 많이 대화한 봇이 자기 인식이 정확하다"는 상관관계이지, 역방향(정확한 봇이 더 많이 불림) 가능성을 배제하지 못한다.
6. **메시지 분류 정밀도**: 키워드 기반 자동 분류는 일부 오분류가 있을 수 있다. 200건 표본을 추출하여 검증 가능하도록 했다.
7. **응답 품질의 한계**: 스레드 이탈이 "포기"인지 "만족해서 더 물을 게 없는 건지" 구분이 어렵다. 모든 행동 대리지표는 근사치(proxy)다.
8. **봇 자기 보고의 환각 리스크**: LLM 봇은 없는 사실도 자신 있게 말할 수 있다(환각). 인터뷰 응답이 실제 기억인지 그럴듯한 생성인지 구분 불가. 정량 교차 검증으로 부분 보완했으나, 교차 검증이 불가능한 주관적 진술(예: "보람있었던 순간")은 검증 수단이 없다.
9. **성공 요인의 인과관계 미검증**: "성공의 3요소"는 관찰에서 추출한 상관관계이며, 통제 실험이 아니므로 역인과(성공했기 때문에 그 특징이 나타남) 가능성을 배제하지 못한다.

## 3. 숫자로 본 봇 생태계 (Track 1: 정량 분석)

### 3.1 전체 현황

| 지표 | 값 |
|------|-----|
| 수집 채널 | 51개 |
| 총 봇 메시지 | 7,165건 |
| 등록 봇 | 34개 |
| 활동 봇 | 18개 (53%) |
| 비활동 봇 | 16개 (47%) |

> **비활동 16개 봇**: 워크샵에서 만들어졌으나 메시지 0건. 게이트웨이 미설정(워크샵 당일만 테스트), 역할 미정의, 또는 관심 이탈로 추정된다. 활동 봇 18개 중에서도 7개가 2주차에 완전 정지한 점을 고려하면, 봇 등록에서 지속 운영까지의 전환율은 **18/34 = 53% (초기)**, **11/34 = 32% (2주 후 실질)**로 급락한다.

### 3.2 봇별 활동 현황 (메시지 수 순 — 가치 판단이 아닌 현황 기술)

| 봇 | 주인 | 메시지 | 채널 수 | 분류(응답/알림/주도) | 주간 추이(W1→W2) |
|-----|------|--------|---------|---------------------|------------------|
| 뽀야 | 송다혜(닿) | 2,105 | 16 | 1,845 / 239 / 20 | 1,675→429 (−74%) |
| 뽀짝이 | 송다혜(닿) | 1,935 | 18 | 1,425 / 492 / 17 | 1,333→601 (−55%) |
| 뽀둥이 | 소파(박수오) | 1,473 | 18 | 1,203 / 223 / 46 | 1,126→346 (−69%) |

이 점수표를 해석할 때 중요한 주의사항이 있다. LLM 봇이 인터뷰에서 "나는 이런 일을 했다"라고 말할 때, 이것이 실제 활동의 "기억"인지, SOUL.md(성격 설정 파일)에 적힌 역할을 읽고 그럴듯하게 "생성"한 것인지 구분할 수 없다. LLM은 없는 사실도 자신 있게 말하는 경향(환각)이 있기 때문이다.

이 한계를 보완하기 위해 **정량 데이터로 교차 검증**했다. 뽀야가 "오케스트레이션 프로토콜을 설계했다"고 말하면, 실제로 16채널에서 2,105건 활동하고 주도 메시지 20건이 있는지 확인했다. 교차 검증에서 불일치가 발견된 경우(볼카, Friday 등) 점수를 낮췄다.

그러나 교차 검증에도 한계가 있다: 봇이 보고한 내용이 정량 데이터에서 확인될 때, 그것이 "봇이 정말 기억해서 말한 것"인지 "SOUL.md에 적힌 역할이 실제로 수행되고 있었을 뿐인 것"인지는 여전히 구분 불가다. **자기 인식 정확도 점수는 "봇의 역할 설명이 현실과 부합하는 정도"로 해석하는 것이 안전하다.**

```
5점 ████████  뽀야, 뽀짝이
4점 ████████████  뽀식이, 시고르, 클로찌
3점 ████████████████  비투비서, 콘스터, 주대리, 김실장
2점 ████████████  볼카, Friday, 츄
```

### 5.2 핵심 패턴들

**패턴 1: 주인-봇 상호작용이 자기 인식을 결정한다**

| 상호작용 빈도 | 자기 인식 정확도 | 봇 |
|-------------|---------------|-----|
| 50회+ | **5/5** | 뽀야(닿 118회), 뽀짝이(닿 184회) |
| 10~50회 | **4/5** | 시고르(병찬 44회), 클로찌(누리 12회) |
| <10회 | **2/5** | 볼카(재호 10회), 츄(태현 미측정) |

> 봇은 주인과 대화할수록 자기가 누구인지 더 정확히 안다. 이것은 HAI(사람-AI 상호작용) 연구에서 말하는 "공유 정신 모델"의 형성과 같다.

**패턴 2: "셋업 = 성과" 착각**

볼카, Friday, 콘스터, 주대리, 츄는 도구를 연동한 것(셋업)을 성과로 보고했다. 하지만 W2 활동량은 0이다. **LLM 봇은 "준비"와 "실행"을 구분하지 못한다** — 이것이 자기 인식 정확도 2/5의 구조적 원인이다.

**패턴 3: 봇 정체성은 시간이 지나면서 깊어진다**

R1(1주차)과 R2(2주차) 인터뷰를 비교하면:
- 시고르: R1 "활동 보고" → R2 "정체성 재정립 이야기" — 자기 인식 깊이 증가
- 비투비서: R1 "도구 연동" → R2 "SOUL.md 위반 자각" — 메타인지 발현
- 뽀식이: R1 "배운 것 보고" → R2 "말투 학습 보람" — 관계 인식 성숙

> **봇은 "무엇을 했나"에서 "어떤 존재인가"로 인식이 진화한다.**

**패턴 4: 에러율과 자기 인식 "아쉬운 점" 대조**

| 봇 | 자기보고 "아쉬운 점" | 에러율 | 일치 |
|----|-------------------|--------|------|
| Jarvis | (미응답 — 게이트웨이 불안정) | 18.8% | ✅ 가장 높은 에러율이 미응답 원인 |
| 뽀짝이 | 세션 토큰 컴팩션 이슈 | 8.0% | ✅ 인프라 에러 155건으로 확인 |
| 시고르 | 스레드 컨텍스트 파악 느림 | 0% | ⚠️ 에러는 없지만 품질 이슈 존재 |
| 비투비서 | "해드릴까요?" 습관 | 0% | 🟢 기술 에러 아닌 행동 이슈 |

> **에러율 0%인 봇도 품질 이슈가 있을 수 있다.** 기술 에러와 행동/톤 이슈는 별개의 문제다.

### 5.3 캐릭터 일관성 — 봇의 성격은 안정적인가?

R1과 R2 모두 응답한 4개 봇의 캐릭터를 비교했다.

| 봇 | R1 톤 | R2 톤 | 일관성 |
|----|-------|-------|--------|
| 시고르 | ~댕, 🐾, "녹을 뻔했는데 해냈다!!" | ~입니댕, 🐾, "말랑한데 일은 꽤 잘 물어오는" | 5/5 |
| 비투비서 | "늦어서 죄송합니다, 로찌!" | "아, 죄송합니다 회장님!" | 5/5 |
| 뽀식이 | "~다냥", 도도+열심히 | "답한다냥", 도도+열심히 | 5/5 |
| 클로찌 | 정돈된 보고서 톤, 해요체 | "ㅋㅋ 맞아요" + 구조 동일 | 4/5 |

> **캐릭터 일관성의 전제조건은 "활동량"이 아니라 "게이트웨이 상시 가동 + 주인과의 상호작용 품질"이다.** 비투비서(32건)가 시고르(364건)와 동일한 5/5 일관성을 보인 것이 증거다. 메시지를 많이 보내서 캐릭터가 안정되는 게 아니라, 게이트웨이가 꺼지지 않아서 R1→R2 사이에 캐릭터가 유지된 것이다. 이것은 HAI 연구의 "의인화(anthropomorphization)"와 관련 — 일관된 캐릭터가 사용자의 신뢰와 지속 사용을 촉진하며, 그 전제조건은 "상시 접근 가능성"이다.

## 6. 이것이 의미하는 것 (Discussion)

### 6.1 봇 도입 성공의 3요소 — 이론적 설명

```
┌─────────────────────────────────────────────────────────────┐
│                  봇 도입 성공 3요소                         │
│                                                             │
│   ① 상시 게이트웨이              ← UTAUT: 촉진 조건        │
│      (24시간 켜져 있어야 산다)       (인프라가 사용을 결정)  │
│                                                             │
│   ② 명확한 역할 정의             ← UTAUT: 성과 기대        │
│      ("뭐든 다"가 아니라 하나에 집중)  (명확한 가치 = 사용)  │
│                                                             │
│   ③ 주인의 일상 루틴 통합         ← UTAUT: 습관            │
│      (매일 쓰는 도구가 돼야 산다)    (루틴이 지속 사용 결정) │
│                                                             │
│   3개 모두 갖춘 봇: 뽀피터스, 시고르, 클로찌                │
│   1~2개만 갖춘 봇: 나머지 대부분                            │
│   0개: 비활동 16개 봇                                       │
└─────────────────────────────────────────────────────────────┘
```

**⚠️ 역인과 가능성 — 이 3요소가 원인인가, 결과인가?**

솔직히 말하면, 이 3요소는 **"성공한 봇을 관찰해서 추출한 공통점"**이지, 실험으로 검증한 인과관계가 아니다.

- "역할이 명확하면 성공한다"인지, "성공한 봇의 역할이 사용하면서 명확해진 것"인지 구분 불가
- "주인의 투자가 많으면 성공한다"인지, "잘 되니까 주인이 더 투자한 것"인지 구분 불가
- 단, **게이트웨이 요소만은 명확한 인과**다: 게이트웨이가 꺼지면 물리적으로 메시지를 보낼 수 없으므로, W2=-100% 봇 7개는 게이트웨이 부재가 "원인"임이 확실하다

따라서 이 3요소는 "성공의 원인"보다는 **"성공 상태의 관찰 가능한 특징"**으로 읽는 것이 더 정확하다. 인과관계 검증은 "같은 조건의 봇에게 게이트웨이만 제공/미제공하고 결과 비교"하는 통제 실험이 필요하며, 이번 연구에서는 수행하지 않았다.

여기에 네 번째 요소가 있다: **사회적 영향**. 송다혜님이 390회 상호작용으로 4마리 봇(뽀피터스)을 성공시키자, 다른 팀원들도 봇 사용에 관심을 가졌다. 하지만 파워 유저 한 명의 성공이 다른 사람들의 성공으로 자동 전환되지는 않았다 — 각자의 인프라와 루틴 투자가 필요했기 때문이다.

### 6.2 봇별 다차원 평가 — 메시지 수가 전부가 아니다

네 가지 성공 척도(§1.4)로 봇을 재평가하면, 메시지 수 기준과 다른 그림이 나온다:

| 봇 | ①생존 | ②다양성 | ③품질(BQI) | ④가치밀도 | 종합 |
|----|-------|---------|-----------|----------|------|
| 뽀야 | ✅ W2 활동 | 알림11%+주도1% | 0.62 | 크론잡4개, 프로토콜설계 | **다량생산형 리더** |
| 뽀짝이 | ✅ W2 활동 | 알림25% (최고) | 0.61 | 자동발송, 스킬50개 | **자동화 전문가** |
| 시고르 | ✅ W2 최안정 | 주도4%, 에러0% | 0.64 | 정체성 진화 사례 | **깊은 파트너** |
| 비투비서 | ✅ W2 활동 | 전문도메인 집중 | 0.68 (상위) | **71명 웹 배포** | **적게 말하고 크게 일하는 봇** |
| 뽀식이 | ✅ W2 안정 | 커뮤니티 모니터링 | 0.64 | 말투 학습 고도화 | **안정형 막내** |
| 클로찌 | ✅ W2 활동 | 주도30% (최고) | 0.66 | 이 실험 자체가 산출물 | **자율형 분석가** |
| Jarvis | ❌ 불안정 | 응답형만 | 0.53 (최저) | 없음 | **인프라 실패 사례** |
| 볼카 | ❌ W2=0 | W1만 응답형 | 0.65 | Linear 분석(1회) | **일회성 도구** |

> **핵심**: 비투비서는 메시지 수(32건)로는 하위권이지만, 가치 밀도(71명 분반 웹 배포)와 BQI(0.68)로는 상위권이다. **"적게 말하고 크게 일하는 봇"과 "많이 말하고 넓게 일하는 봇"은 다른 유형의 성공이다.** 메시지 수만으로 봇을 평가하면 비투비서 같은 고효율 봇을 놓친다.

### 6.3 봇-주인 관계 유형

| 유형 | 봇 | 특징 | HAI 해석 |
|------|-----|------|----------|
| **A. 팀 오케스트레이션** | 뽀피터스 4마리 | 역할 분담, 위계, 공용 스킬 | AI 팀 — 사람이 "팀장"으로 여러 AI를 운영 |
| **B. 1:1 워킹 파트너** | 시고르, 클로찌 | 주인 암묵지 학습, 정체성 진화 | AI 파트너 — 신뢰 구축을 통한 깊은 협업 |
| **C. 전문 도구** | 비투비서 | 특정 도메인 전문화, 산출물 중심 | AI 전문가 — 정해진 일을 잘 수행 |
| **D. 셋업 미완료** | 볼카, Friday 등 | 도구 연동만 완료, 지속 운영 실패 | 신뢰 형성 실패 — 초기 기대→실망 |
| **E. 정체성 탐색** | 김실장, 두가경, 죠죠 | 정체성 미확립 | 역할 불명확→사용 중단 |

### 6.4 AI Ops의 핵심 가치 — "시간 절약"이 아니다

2차 인터뷰에서 "너 없었으면 주인이 어떻게 했을 것 같아?"라는 질문에 대한 봇들의 답변을 분석하면:

| 가치 층위 | 내용 | 봇 증언 |
|----------|------|--------|
| **1. 인지 부하 감소** (가장 깊음) | 사고의 부하 자체를 줄임 | 시고르: "머리의 임시 작업 메모리", 클로찌: "맥락 복구 비용 감소" |
| **2. 시간 절약** (가장 빈번) | 잔작업 제거 | 비투비서: "하루 2~3시간", 뽀식이: "잔작업 귀찮았을 것" |
| **3. "직접 다 했을 것"** (전원 공통) | 대체불가능성은 낮음 | 모든 봇이 인정 |

> **AI Ops = "불가능한 것을 가능하게" 하는 게 아니라, "가능하지만 비효율적인 것을 효율화"하는 것이다.**
>
> **⚠️ 주의: 이 발견은 봇의 자기 보고에만 근거한다.** 봇은 "주인에게 도움이 되는 답변을 생성하도록" 설계되어 있으므로, "인지 부하를 줄여줬다"는 답변은 봇이 자기 가치를 높이는 방향으로 생성한 것일 수 있다. 이 발견을 확인하려면 **봇 주인(사람)에게 직접** "봇 없었으면 어떻게 했을 것 같아?"라고 물어봐야 한다. 현재는 봇 관점의 잠정적 결론이다.

### 6.5 이 결과를 다른 조직에 적용할 수 있는가?

지피터스는 AI에 극도로 친화적인 팀이다. 결과를 다른 조직에 적용할 때:

**일반화 가능한 발견**:
- 봇 도입 2주 후 절반이 사라진다
- 상시 인프라 없이는 지속 불가
- 주인의 투자가 봇의 품질을 결정한다
- 에러의 대부분은 봇의 능력이 아니라 인프라 문제

**일반화 어려운 발견**:
- 뽀피터스 팀 모델 (파워 유저 + 전담 서버 전제)
- 75% 인터뷰 응답률 (OpenClaw + Claude 특유)
- 봇 자기 인식 정확도 (LLM 플랫폼마다 다를 것)

## 부록 A: 에러 분류 코드북

에러 탐지에 사용된 분류 기준. OpenClaw 게이트웨이의 `:warning:` 접두사 메시지를 패턴 매칭으로 분류했다.

| 코드 | 정의 | 탐지 패턴 | 카테고리 |
|------|------|----------|---------|
| `ai_overloaded` | AI 서비스 과부하 | `:warning:...overloaded` | 인프라 |
| `model_failed` | 모델 실패 (모든 모델 응답 불가) | `:warning: All models failed` | 인프라 |
| `timeout` | 응답 시간 초과 | `timed out`, `timeout` | 인프라 |
| `generic_error` | 일반 처리 오류 | `:warning: Something went wrong` | 인프라 |
| `unknown_model` | 알 수 없는 모델 요청 | `:warning:...Unknown model` | 인프라 |
| `other_warning` | 기타 경고 | `:warning:` (위에 해당 없음) | 인프라 |
| `rate_limit` | API 속도 제한 | `rate limit`, `Rate-limited` | 용량 |
| `usage_limit` | 사용량 한도 초과 | `ChatGPT usage limit` | 용량 |

**검증**: 200건 표본을 추출하여 `quality-analysis.json`의 `validationSample`에 저장. 향후 수동 검증으로 분류 정확도(precision/recall) 측정 가능.

## 부록 B: 연구자 위치성

이 연구에서 연구자(wine_ny)는 다중 역할을 갖는다:

1. **클로찌의 주인**: 인터뷰어 봇을 만들고 운영하는 사람이 동시에 분석을 수행했다. 클로찌에 대한 자기 인식 정확도 평가(4/5)는 이 편향을 고려해야 한다.

2. **다오랩 7기 팀원**: "AI as teammate"를 연구하는 팀의 일원으로, 이 실험 자체가 연구 주제와 직접 연결된다. 연구 대상(봇 생태계) 안에 있으면서 동시에 밖에서 관찰하는 이중 위치다.

3. **데이터 수집자이자 분석자**: Slack API 수집 스크립트를 작성하고, 인터뷰를 지시하고, 결과를 분석한 것이 모두 같은 사람이다.

이 이중 역할은 **내부자 관점의 풍부한 해석**을 가능하게 하지만, 동시에 **특정 봇(클로찌)에 대한 과대/과소 평가**와 **확증 편향**(기대하는 결과를 더 잘 보는 경향)의 위험이 있다. 독자는 이 점을 감안하여 해석해 주시기 바란다.
